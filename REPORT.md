# 视觉媒体通信大作业：高性能 YUV 视频播放器实践报告

## 一、 项目目标

本次大作业的目标是开发一个能够播放无压缩 YUV 格式视频的播放器。作为一名刚接触视频处理的大学生，我的主要任务不仅是把视频显示出来，还要探索如何让它“跑得更快”。

具体要求包括：
1.  **基础功能**：使用 C 语言读取 YUV420P 格式的视频文件，并将其转换为 RGB 格式，最后在屏幕上播放。
2.  **性能优化（必做）**：
    *   **SSE 指令集加速**：利用 CPU 的 SIMD（单指令多数据）能力，一次处理多个像素，提升颜色转换的效率。
    *   **多线程处理**：利用多核 CPU，把一帧图像分成几块同时处理，进一步榨干电脑性能。
3.  **跨平台界面（必做）**：使用 Qt 框架（C++）来制作播放器的界面，因为它在不同系统上都能跑，而且看起来比较现代。
4.  **进阶功能（扩展）**：
    *   支持 Y4M 格式的自动解析。
    *   支持倍速播放（0.5x - 4.0x）。
    *   支持快进/快退（±15帧）。
    *   支持个性化水印。

## 二、 实现方案

为了完成这个项目，我采用了“C 语言写核心算法，C++ 写界面”的混合编程方式。这样既能保证核心代码的高效（C 语言），又能利用 Qt 的便捷性（C++）。

### 2.1 整体架构

项目主要分为两个部分：
*   **Core（核心层）**：负责干脏活累活。包括读取视频文件、把 YUV 数据算成 RGB 数据。这部分全是 C 语言写的，为了追求速度。
*   **GUI（界面层）**：负责好看。使用 Qt 框架，负责显示画面、响应按钮点击（播放、暂停、打开文件）。

### 2.2 关键模块设计

1.  **YUV 读取模块 (`yuv_reader.c`)**：
    *   负责打开 `.yuv` 或 `.y4m` 文件。
    *   **智能解析**：对于 `.y4m` 文件，能够自动读取文件头，解析出视频的宽（W）和高（H），避免了手动输入的麻烦。
    *   **鲁棒性**：正确处理了文件头和数据偏移，解决了横向花屏和内存对齐问题。

2.  **颜色转换模块 (`yuv_converter.c` / `sse.c`)**：
    *   这是最耗时的部分。原始公式涉及浮点运算，我先用整数移位优化了标量版本（比如用 `x * 1436 >> 10` 代替 `x * 1.402`）。
    *   **SSE 优化**：使用 `_mm_loadu_si128` 一次加载 16 个像素，用 `_mm_mullo_epi16` 并行计算，大大减少了循环次数。

3.  **多线程模块 (`thread_pool.c`)**：
    *   使用 Windows 原生 API (`_beginthreadex`)。
    *   把一帧图像按高度切成 4 块（假设是 4 核 CPU），每块启动一个线程去跑 SSE 转换函数。
    *   主线程等待所有子线程干完活，再把结果拿去显示。

### 2.3 计算复杂度分析

重点关注 **YUV 转 RGB** 这一核心模块，我们从理论计算量和实际优化两个角度进行分析。

假设视频分辨率为 $W \times H$，总像素数为 $N = W \times H$。

1.  **标量版本 (Scalar)**
    *   **算法逻辑**：对于每个像素，需要进行 5 次乘法、4 次加/减法和 3 次移位操作（优化后）。
    *   **计算量**：
        *   乘法：$5N$
        *   加减法：$4N$
        *   移位/逻辑：$3N$
        *   **总操作数**：约 $12N$ 次基本运算。
    *   **复杂度**：$O(N)$，即与图像像素数成正比。虽然是线性复杂度，但系数较大，且无法利用 CPU 的并行能力。

2.  **SSE 指令集版本 (SIMD)**
    *   **算法逻辑**：SSE 寄存器为 128 位，可以同时容纳 16 个 8 位整数（`uint8_t`）或 8 个 16 位整数（`int16_t`）。由于中间计算需要防止溢出，我们通常将数据解包为 16 位进行计算。这意味着一条指令可以同时处理 **8 个像素**。
    *   **计算量**：
        *   理论上，指令数减少为原来的 $1/8$。
        *   实际上，由于需要额外的 `load`（加载）、`unpack`（解包）、`pack`（打包）和 `store`（存储）指令，加速比通常达不到 8 倍。
    *   **复杂度**：仍然是 $O(N)$，但常数系数大幅降低。
    *   **优势**：利用 CPU 的超标量流水线和宽位宽寄存器，显著提升吞吐量。

3.  **多线程版本 (Multi-threading)**
    *   **算法逻辑**：将图像按行划分为 $M$ 个块（例如 $M=4$），每个线程处理 $N/M$ 个像素。
    *   **理论加速比**：接近 $M$ 倍（理想情况下）。
    *   **实际瓶颈**：
        *   **内存带宽**：所有线程共享内存总线，如果计算速度超过了内存读取速度，加速比会受限（Memory Bound）。
        *   **线程开销**：创建和销毁线程需要时间。对于小分辨率（如 CIF 352x288），线程开销可能大于计算收益；但对于高清（1080p+），计算量占主导，加速效果极其明显。

4.  **实际测试验证**
    *   在 1080p 分辨率下，实测加速比达到了 **4.60x**（SSE + 4线程 vs 标量）。
    *   这验证了：SIMD 提供了单核内的指令级并行（约 1.5x），而多线程提供了多核间的任务级并行（约 3x），两者叠加产生了显著的性能飞跃。

## 三、 实现结果与性能分析

### 3.1 功能验证

经过开发与调试，播放器完整实现全部设计目标，能够稳定打开并播放标准 .yuv 与 .y4m 格式测试视频，画面清晰无失真，色彩还原准确。
播放控制操作响应及时，倍速切换、帧跳转等功能流畅稳定，界面布局合理，水印、状态信息显示正常，整体达到可用、易用的实践要求。

### 3.2 发现与调优 (Discovery & Optimization)

**现象**：
在测试过程中，我发现了一个反直觉的现象：在播放 CIF 分辨率（352x288，如 `foreman.yuv`）的小视频时，开启 **多线程 (Multi-threading)** 模式后的处理耗时（约 1.3ms）竟然比单线程的 **SSE** 模式（约 0.2ms）还要慢！

**原因分析**：
经过查阅资料和分析，这是由于 **线程开销（Thread Overhead）** 导致的。
*   对于小图（总像素约 10万），单线程处理极快（微秒级）。
*   此时，创建线程、分发任务、上下文切换（Context Switch）以及线程销毁的开销反而成为了主要耗时。
*   这就好比“杀鸡用牛刀”，准备工作比实际干活还累。

**解决方案：智能策略 (Auto Mode)**：
为了解决这个问题，我设计并实现了一个 **自动选择最优模式 (Auto)** 的功能。
*   **策略逻辑**：程序在加载视频时，会根据视频的总像素数进行判断。
    *   **如果像素数 < 40万**（如 CIF）：自动选择 **SSE 模式**，避免多线程开销。
    *   **如果像素数 >= 40万**（如 720p/1080p）：自动选择 **多线程模式**，利用并行计算优势。
*   **效果**：通过这种启发式策略，播放器在任何分辨率下都能始终保持最佳性能，无需用户手动切换。

### 3.3 性能对比（SSE 与多线程的威力）

为了直观地展示优化的效果，我编写了一个专门的测试程序 (`src/benchmark.cpp`)，在 1920x1080 分辨率下，分别运行 100 次转换，计算平均耗时。

测试环境：
*   CPU: 4 核处理器
*   分辨率: 1920x1080 (1080P)
*   循环次数: 100 次
*   编译模式: **Release** (这一点至关重要，Debug 模式下优化效果不明显)

| 模式 | 平均每帧耗时 (ms) | 相对标量加速比 | 备注 |
| :--- | :---: | :---: | :--- |
| **标量 (Scalar)** | 6.61 ms | 1.00x | 基础 C 语言实现 (已做整数优化) |
| **SSE 指令集** | 4.24 ms | 1.56x | 开启 SIMD 优化 |
| **多线程 (MT)** | 1.44 ms | **4.60x** | 4 线程 + SSE |

## 四、 遇到的问题与解决

在开发过程中，我踩了不少坑，但也学到了很多：

1.  **Debug vs Release 性能差异**：
    *   *问题*：一开始测试时发现多线程反而比单线程慢，SSE 也没快多少。
    *   *解决*：后来发现是编译模式的问题。Debug 模式下编译器不做优化且包含大量调试信息。切换到 Release 模式后，性能直接起飞。

2.  **Y4M 格式解析错误**：
    *   *问题*：播放 1080p 的 `.y4m` 文件时出现严重的花屏（横向条纹）和卡死。
    *   *解决*：
        *   卡死是因为内存分配时用了旧的分辨率（352x288）去分配 1080p 的数据，导致堆溢出。我修正了 buffer 大小的计算逻辑。
        *   花屏是因为 `yuv_reader` 返回的结构体中 `width` 字段没有更新，导致 GUI 按照错误的分辨率去渲染。我确保了在解析文件头后正确更新所有相关字段。

3.  **播放器逻辑 Bug**：
    *   *问题*：点击播放按钮没有反应，或者只有在切换倍速时才开始播放。
    *   *解决*：这是因为定时器的初始化逻辑有漏洞。我在 `playPause` 函数中确保了在启动定时器前先设置播放状态，并强制计算正确的定时间隔。

4.  **CMake 版本报错**：
    *   *问题*：Qt 6 要求 CMake 3.22+，但我电脑上只有 3.20。
    *   *解决*：在 `CMakeLists.txt` 里加了一句 `set(QT_FORCE_MIN_CMAKE_VERSION_FOR_USING_QT 3.20)` 强制让它运行。

## 五、 心得体会

这次大作业让我从零开始完成了一个“高性能”软件。
我最大的收获是理解了**“优化”**的意义。以前写代码只求“能跑”，现在知道原来同样的功能，写法不同，性能可以差好几倍。看着 1080P 的视频在自己写的播放器里流畅播放，而且 CPU 占用率还很低，这种成就感是无与伦比的。

同时，我也熟悉了 CMake 构建系统和 Qt 框架，这对我以后的开发工作非常有帮助。虽然中间报错的时候很抓狂（特别是那个横向花屏的问题），但解决问题后的快感也是编程最迷人的地方。
